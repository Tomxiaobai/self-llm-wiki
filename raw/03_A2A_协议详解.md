# A2A 协议详解

> A2A（Agent2Agent）是 Google 贡献给 Linux Foundation 的开放协议，让不同框架、不同公司构建的 AI Agent 能够互相发现、通信与协作。
> 协议地址：https://github.com/google/A2A
> 许可证：Apache 2.0

---

## 1. 核心设计目标

A2A 解决的核心问题：**当 Agent A 需要委托 Agent B 执行某任务时，如何跨框架、跨服务、安全地完成协作？**

四大能力：

| 能力 | 说明 |
|------|------|
| **能力发现** | 通过 Agent Card 了解对方能做什么 |
| **模态协商** | 协商交互方式（文本/文件/结构化数据）|
| **长任务协作** | 安全执行耗时较长的异步任务 |
| **不透明性** | 无需暴露内部状态、记忆或工具实现 |

---

## 2. 架构概览

```
A2A Client                    A2A Server（目标 Agent）
    │                               │
    │  1. 获取 Agent Card            │
    │◄──────────────────────────────│ /.well-known/agent.json
    │                               │
    │  2. 发送任务（JSON-RPC 2.0）    │
    │──────────────────────────────►│
    │                               │
    │  3. 接收流式进度（SSE）         │
    │◄──────────────────────────────│
    │                               │
    │  4. 异步推送通知（Webhook）      │
    │◄──────────────────────────────│
```

### 通信层

| 绑定方式 | 协议 |
|---------|------|
| 主要 | JSON-RPC 2.0 over HTTPS |
| 可选 | gRPC（通过 a2a.proto）|
| 可选 | HTTP+JSON REST |

## 3. Agent Card（代理名片）

Agent Card 是 A2A 的**能力发现机制**，发布在 `/.well-known/agent.json`：

```json
{
  "name": "RAG Knowledge Agent",
  "description": "专业知识库检索与问答 Agent",
  "url": "https://my-agent.example.com",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true,
    "extendedAgentCard": false
  },
  "skills": [
    {
      "id": "knowledge-qa",
      "name": "知识库问答",
      "description": "基于内部文档库回答问题",
      "inputModes": ["text"],
      "outputModes": ["text"]
    }
  ],
  "defaultInputModes": ["text"],
  "defaultOutputModes": ["text"],
  "securitySchemes": {
    "bearerAuth": { "type": "http", "scheme": "bearer" }
  }
}
```

## 4. Task 状态机

Task 是 A2A 的核心有状态单元，生命周期如下：

```
           ┌─────────────┐
           │  submitted  │  ← 任务已接收，等待处理
           └──────┬──────┘
                  │
           ┌──────▼──────┐
           │   working   │  ← Agent 正在处理
           └──────┬──────┘
                  │
        ┌─────────┼─────────┐
        │                   │
┌───────▼───────┐   ┌───────▼──────┐
│ input-required│   │ auth-required│  ← 等待客户端输入/授权
└───────┬───────┘   └───────┬──────┘
        └─────────┬─────────┘
                  │ 客户端响应后继续
                  │
      ┌───────────┼───────────┐
      │           │           │
┌─────▼────┐ ┌───▼───┐ ┌─────▼────┐
│completed │ │failed │ │ canceled │  ← 终态
└──────────┘ └───────┘ └──────────┘
      也可能为: rejected（Agent 拒绝处理）
```

**终态**（不可再接收消息）：`completed`、`failed`、`canceled`、`rejected`

## 5. 核心数据结构

### Task 对象

```json
{
  "id": "task-uuid-123",
  "contextId": "session-uuid-456",
  "status": {
    "state": "working",
    "message": null,
    "timestamp": "2025-03-26T10:00:00Z"
  },
  "history": [ ],
  "artifacts": [ ],
  "metadata": { }
}
```

> 注意：Task ID 由**服务端生成**，客户端无法指定。

### Message 对象

```json
{
  "role": "user",
  "parts": [
    { "kind": "text", "text": "帮我分析这份报告" },
    { "kind": "file", "file": { "mimeType": "application/pdf", "bytes": "..." } }
  ],
  "contextId": "session-uuid-456",
  "taskId": "task-uuid-123"
}
```

### Part 类型

| 类型 | kind 值 | 描述 |
|------|---------|------|
| 文本 | `text` | 纯文本内容 |
| 文件 | `file` | 文件内容（Base64 或 URL）|
| 结构化数据 | `data` | JSON 对象 |

## 6. 核心 API 方法

| 方法 | 返回 | 说明 |
|------|------|------|
| `message/send` | Task 或 Message | 发送消息，默认阻塞等待结果 |
| `message/stream` | SSE 流 | 流式接收实时进度 |
| `tasks/get` | Task | 轮询任务状态 |
| `tasks/list` | Task[] | 分页列出任务 |
| `tasks/cancel` | Task | 取消任务（幂等）|
| `tasks/resubscribe` | SSE 流 | 重新订阅已有任务的流 |

---

## 7. 流式响应（SSE）

当 Agent Card 中 `capabilities.streaming: true` 时，客户端可使用流式 API：

```
client → POST /message/stream

server → SSE events:
  data: {"type": "TaskStatusUpdate", "status": {"state": "working"}}
  data: {"type": "TaskArtifactUpdate", "artifact": {"parts": [{"kind": "text", "text": "正在检索..."}]}}
  data: {"type": "TaskStatusUpdate", "status": {"state": "completed"}}
```

## 8. 多轮对话与 contextId

- `contextId`：会话级别，标识一次对话上下文（跨多个 Task）
- `taskId`：任务级别，标识单次任务执行

```
同一 contextId 下的多轮交互：

Round 1: contextId=C1, taskId=T1 (completed)
Round 2: contextId=C1, taskId=T2 (completed)
Round 3: contextId=C1, taskId=T3 ...

中途需要用户输入（input-required）：
  → 客户端发新消息，携带相同 taskId 和 contextId
  → Agent 恢复执行
```

## 9. A2A vs MCP — 区别与互补

| 维度 | A2A | MCP（Model Context Protocol）|
|------|-----|-----------------------------|
| **定位** | Agent ↔ Agent 通信 | Model ↔ Tool/Resource 集成 |
| **交互对象** | 智能体之间 | 模型与工具/数据源之间 |
| **交互形式** | 任务委派、协作 | 工具调用、上下文注入 |
| **状态暴露** | 不暴露内部实现 | 工具接口对调用方可见 |
| **适用场景** | 多 Agent 协作系统 | 单 Agent 工具增强 |

**互补关系（推荐架构）：**

```
Agent 内部：
  LLM ──[MCP]──► Tool A（搜索）
               ► Tool B（数据库）
               ► Tool C（代码执行）

Agent 之间：
  Agent A ──[A2A]──► Agent B（专业知识库）
                  ► Agent C（图像分析）
                  ► Agent D（代码生成）
```

> 一个 Agent 内部用 MCP 调用工具，对外通过 A2A 与其他 Agent 协作。

## 11. 在 Eino 中集成 A2A

Eino Agent 作为 A2A Server 对外暴露能力的基本思路：

```go
// 1. 构建 Eino Agent（内部逻辑）
agent := buildReActAgent(ctx, chatModel, tools)

// 2. 包装为 A2A Server
// Agent Card 描述该 Agent 的能力
agentCard := a2a.AgentCard{
    Name:        "Eino RAG Agent",
    Description: "基于 Eino 构建的 RAG 问答 Agent",
    URL:         "https://my-service.example.com",
    Capabilities: a2a.Capabilities{
        Streaming: true,
    },
    Skills: []a2a.Skill{
        {ID: "qa", Name: "知识问答", InputModes: []string{"text"}},
    },
}

// 3. 处理 A2A 请求 → 转发给 Eino Agent
http.HandleFunc("/message/send", func(w http.ResponseWriter, r *http.Request) {
    // 解析 A2A 请求 → 转为 Eino 输入 → 调用 agent.Invoke → 包装为 A2A 响应
})
```
