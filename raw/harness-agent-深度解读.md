## 一、论文核心观点：一句话总结

> **Agent 系统的可靠性瓶颈不是模型能力，而是包裹模型的执行基础设施层 — Agent Harness（代理执行挽具）。**

用作者的类比：学术界对"引擎"（模型）研究得很透彻，但对"底盘"（Harness）几乎一无所知。

## 二、什么是 Agent Harness？

### 2.1 定义

Agent Harness 是一个**运行时治理系统** H = (E, T, C, S, L, V)，包含六个核心组件：

```
┌─────────────────────────────────────────────────┐
│           L - 生命周期钩子 (Lifecycle Hooks)        │
│   ┌───────────────────────────────────────────┐  │
│   │         E - 执行循环 (Execution Loop)       │  │
│   │                                           │  │
│   │  C(上下文管理) → LLM → T(工具注册表)         │  │
│   │       ↑                    ↓              │  │
│   │       └──── S(状态存储) ←──┘              │  │
│   └───────────────────────────────────────────┘  │
│         V - 评估接口 (Evaluation Interface)        │
└─────────────────────────────────────────────────┘
```

| 组件 | 全称 | 功能 | 对应的失败模式 |
|------|------|------|-------------|
| **E** | Execution Loop | 管理 观察-思考-行动 循环、终止条件、错误恢复 | 执行失控 (runaway) |
| **T** | Tool Registry | 维护工具接口目录，路由和监控工具调用 | 工具误用 |
| **C** | Context Manager | 控制每一轮进入模型上下文窗口的信息 | 上下文膨胀 (blowout) |
| **S** | State Store | 跨轮次/跨会话持久化任务状态，支持故障恢复 | 状态丢失 |
| **L** | Lifecycle Hooks | 认证、日志、策略执行的前/后拦截点 | 未监控的副作用 |
| **V** | Evaluation Interface | 捕获动作轨迹、中间状态、成功信号用于离线分析 | 不可观测的行为 |

### 2.2 最小条件 vs 充分条件

- **必要条件**: 至少实现 E + T（没有 E 就没有多步执行；没有 T 就无法作用于外部世界）
- **充分条件**: 六个组件全部以生产级可靠性实现 → **全栈 Harness**

### 2.3 边界案例判定

| 系统 | 分类 | 理由 |
|------|------|------|
| **ReAct** | 框架原语 (非 Harness) | E 仅为 while 循环，无错误恢复；T 为临时文本解析，无注册表 |
| **AutoGPT** | 单体 Harness | E/T/S 紧耦合但都存在；可靠性问题源于实现质量而非架构缺失 |
| **LangGraph** | 拓扑编码 Harness | DAG 结构保证 E 的活性；但 C 是隐式的（通过图拓扑实现） |
| **MemGPT** | 能力模块 | C 和 S 实现精湛，但 E 和 T 委托给宿主环境 |
| **AIOS** | 全栈 Harness | 六个组件全部以显式 OS 级抽象实现 |

## 三、为什么 Harness 是"绑定约束"？

### 3.1 六项独立实证研究的收敛证据

**核心发现：仅改变 Harness（不改变模型），就能获得巨大性能提升。**

| 研究 | 变化内容 | 效果 | 模型是否改变？ |
|------|---------|------|-------------|
| **Pi Research** (can1357) | 仅改变编辑工具格式 | 6.7% → 68.3% (**10倍**) | 否 |
| **LangChain DeepAgents** | Harness层变更 | 52.8% → 66.5% (**+26%**) | 否 |
| **Meta-Harness** (Stanford/MIT) | 自动化 Harness 优化 | 超越手工工程 76.4% vs 74.7% | 否 |
| **Meta-Harness 跨模型** | 一个 Harness 五个模型 | +4.7pp 数学推理 | 否 |
| **Meta-Harness 效率** | 上下文管理优化 | +7.7pp，使用 4x 更少 token | 否 |
| **AgencyBench** | 跨 Harness 评估 | 31% 性能方差归因于 Harness 选择 | 显式控制 |

### 3.2 产业界验证

- **OpenAI Codex**: 3-7人团队，5个月，100万行生产代码，零手写代码。瓶颈不是模型能力，而是"环境未充分指定"
- **Stripe Minions**: 每周1300+ PR，"Blueprint"混合架构（确定性节点 + 代理循环）
- **Vercel**: 删除80%可用工具后，任务成功率提升超过任何模型升级

## 四、历史演化：三条谱系的融合

```
软件测试 Harness ──────────┐
  (治理包装模式)              │
                            ├──→ Agent Harness
RL 环境 ──────────────────┤     (2024-2026)
  (接口标准模式)              │
                            │
早期 LLM 框架 ────────────┘
  (失败模式目录)
```

### 三个工程范式的演进

| 时代 | 范式 | 核心问题 | 工程表面 |
|------|------|---------|---------|
| 2022-2024 | **Prompt Engineering** | "给模型什么文本？" | 窄 - 单一文本输入 |
| 2025 | **Context Engineering** | "模型应该看到什么信息？" | 中 - 上下文窗口分配 |
| 2026 | **Harness Engineering** | "需要什么治理、约束和反馈循环？" | 宽 - 完整六组件基础设施 |

每个范式**包含**前一个：Harness Engineering ⊃ Context Engineering ⊃ Prompt Engineering

### 5.2 五大类别

```
Agent Harness 系统
├── 全栈 Harness (E,T,C,S,L,V 全部实现)
│   ├── 通用/生产: DeepAgents, Claude Code, OpenClaw, OpenHands, AIOS
│   └── 专用: SWE-agent, Browser-Use, RAI, PortiaAI
├── 框架 (E,T 实现, C/S/L/V 部分或缺失)
│   ├── 编排框架: LangGraph, Prefect
│   ├── 推理框架: vLLM, SGLang
│   └── 多Agent编排: AutoGen, Google ADK, CrewAI
├── 能力模块 (C,S 精深, 其他最小)
│   ├── 核心模块: MemGPT, MCP Servers, Voyager
│   └── 评估基础设施: HAL, AgencyBench, SkillsBench, Harbor
```

### 5.3 关键发现

- **模块化差距**: 全栈 Harness 大多从零重新实现每个组件，缺少标准可复用模块（S/L/V 尤其缺乏）
- **V 和 L 是最被忽视的组件**: 22个系统中14个的 V 组件为部分或缺失
- **评估基础设施 = 最严格的 Harness 工程**: 因为基准结果必须可复现

## 六、九大核心技术挑战

论文将挑战分为三大主题、九个领域：

### 第一主题：执行基础设施 (§5)

#### 6.1 沙箱与安全

**Agent Harness 面临独特的威胁模型：**

| 攻击向量 | 入口 | 利用的组件 | 持久性 | 严重程度 |
|---------|------|----------|--------|---------|
| 直接提示注入 | 用户输入 | C | 会话内 | 低 |
| 间接注入（检索） | 外部内容 | C | 会话内 | 中 |
| 工具中介注入 | 工具输出 | T | 多步骤 | 高 |
| **记忆投毒** | 持久存储写入 | **S** | **跨会话** | **最高** |
| 能力升级 | 工具组合 | T | 单会话 | 高 |
| 沙箱逃逸 | 执行环境 | E | 系统级 | 极高 |
| 跨Agent注入 | Agent间消息 | E | 多Agent | 高 |

**五种隔离机制对比：**

| 机制 | 启动延迟 | 内存开销 | 隔离强度 | 适用场景 |
|------|---------|---------|---------|---------|
| Process-only | ~10ms | +0 MB | 无 | 仅原型 |
| Docker/OCI | ~1s | +10-50 MB | 命名空间 | 开发/可信Agent |
| gVisor | ~2s | +50-100 MB | 用户态系统调用 | 安全敏感部署 |
| **Firecracker MicroVM** | **~125ms** | **+5-15 MB** | **硬件虚拟化** | **生产多租户** |
| WebAssembly | ~10ms | +1-5 MB | 能力沙箱 | 计算受限任务 |

> SandboxEscapeBench 发现 GPT-4 级模型能以 15-35% 成功率逃逸 Docker 容器 — Docker 隔离在安全性上是不够的。

#### 6.2 评估与基准测试

**评估不可靠的三个根本原因：**

1. **环境漂移** (时间稳定性问题): 真实环境在基准创建和评估之间发生变化
2. **任务规范歧义** (通信完整性问题): 自然语言描述遗漏隐含约束
3. **Harness 耦合** (测量有效性问题): 性能是 (Agent, Harness, 环境) 三元组的属性

> **METR 发现**: 维护者的合并率比 SWE-bench 自动评分低 24.2 个百分点，且差距在扩大而非缩小。

**五项基准设计原则：**
1. 显式 Harness 规范 — 报告完整配置
2. 环境隔离保证 — 可验证的隔离条件
3. 部分信用分解 — 非二元成功/失败
4. 成本校准报告 — 报告每任务成本
5. 行为异常披露 — 检测基准博弈

#### 6.3 协议与接口标准化

**MCP vs A2A：互补而非竞争**

| 维度 | MCP (Anthropic) | A2A (Google) |
|------|----------------|-------------|
| 范围 | Agent→Tool | Agent→Agent |
| 传输 | JSON-RPC (stdio/HTTP) | HTTPS + SSE |
| 延迟 | 2-15ms (本地) | 50-200ms (网络) |
| 有状态会话 | 路线图 (2026) | 原生支持 |
| 工具发现 | ✓ (schema registry) | ~ (能力声明) |
| 多步协调 | × | ✓ |
| **Harness角色** | **T组件标准** | **E组件协调** |

> 三层协议栈：意图(ACP) → 任务委派(A2A) → 工具执行(MCP)

### 第二主题：状态与知识管理 (§6)

#### 6.4 运行时上下文管理

**六种策略对比：**

| 策略 | 机制 | 优点 | 缺点 | 代表系统 |
|------|------|------|------|---------|
| 截断 | 保留最近 K tokens | 简单快速 | 丢失旧信息 | 基线 |
| 摘要 | 压缩旧上下文 | 保留要点 | 信息损失 | LangChain |
| 检索增强 | 全部存储，检索相关 | 保留一切 | 检索质量依赖查询 | MemGPT, Zep |
| 知识图谱 | 结构化实体/关系存储 | 语义关系 | 构建成本 | Zep TK |
| 记忆即动作 | Agent主动管理记忆 | 任务最优策划 | 学习开销 | MemAct |
| 技能注入 | 结构化程序性知识包 | 定向高效 | 需要策划 | SkillsBench |

**长上下文模型不能消除 C 组件的必要性：**
- "Lost in the Middle" 现象：模型在长上下文中对早期信息的有效注意力下降
- 长上下文 → 挑战从"保留"转向"突显"（决定高亮什么，而非保留什么）
- SkillsBench 发现：2-3个模块的精简 Skill 优于完整文档

#### 6.5 工具使用治理

**五种工具注册表设计模式：**

| 模式 | 特点 | 安全性 | 灵活性 |
|------|------|--------|--------|
| 静态声明式 | 初始化时固定工具集 | 最高 | 最低 |
| 动态注册+Schema验证 | 运行时注册，验证后添加 | 高 | 高 |
| 能力范围限定 | 按范围授权（文件/网络/执行） | 高 | 中 |
| 语义搜索注册表 | 向量索引，按需检索 | 中 | 最高 |
| 学习式工具规划 | Harness辅助的搜索树 | 中 | 高 |

> **Vercel 发现**: 将工具从15个减少到2个，准确率从80%提升到100% — 减少工具比增加工具更能提升可靠性。

#### 6.6 记忆管理架构

**记忆治理合约（论文提议的六属性规范）：**

| 维度 | 属性 | 说明 |
|------|------|------|
| 持久性 | 写入持久性 | 写入后在进程/机器重启后存活 |
| 持久性 | 读取持久性 | 后续读取返回符合保留策略的内容 |
| 一致性 | 跨轮次一致性 | 同一任务内的读取反映所有先前写入 |
| 一致性 | 跨会话一致性 | 跨会话的记忆状态符合声明的持久化策略 |
| 隔离与审计 | 内容隔离 | 不同Agent实例的记忆互不可访问 |
| 隔离与审计 | 写入可审计性 | 维护所有写操作的审计日志 |

**记忆即安全攻击面**: 允许Agent控制写入持久存储 = 隐式接受 S 组件的攻击面受限于 Agent 的判断力。

### 第三主题：协调与规划 (§7)

#### 6.7 规划与推理基础设施

**规划复杂度升级 → 治理责任从模型转移到 Harness：**

```
线性规划 (ReAct)          → Harness 责任: 20%, 模型: 80%
  ↓
树结构规划 (ToT/LATS)     → Harness 责任: 60%, 模型: 40%
  ↓
MCTS 规划 (Agent Q/ExACT) → Harness 责任: 85%, 模型: 15%
```

**Harness 必须治理的四个规划功能：**
1. **循环控制**: 触发条件、观察注入策略、中止条件
2. **规划状态持久化**: 跨推理调用维护分支状态
3. **搜索预算执行**: 限制计算消耗，预算耗尽时必须提交计划
4. **ACI 接口设计**: Agent-Computer Interface 完全由 Harness 决定

> SWE-agent 发现：ACI 设计对规划性能的可测量影响大于模型能力。

#### 6.8 多Agent协调

**四种多Agent编排模式：**

| 模式 | 身份模型 | 消息验证 | 状态一致性需求 | 关键脆弱性 |
|------|---------|---------|-------------|----------|
| **角色编排** (MetaGPT) | 固定SOP分配 | 类型化文档交接 | 原子文档级 | 级联错误 |
| **市场协调** (AutoGen) | 动态注册表 | 出价/报价 | Agent注册表可用性 | 任务饥饿 |
| **模拟基底** (Generative Agents) | 持久实体 | 通过环境状态间接 | 世界模型冲突解决 | 共享状态损坏 |
| **层级委派** (DeerFlow) | 权限树 | 任务规范+结果验证 | 权限状态传播 | 权限升级 |

**关键负面结果**: 精心设计的单Agent系统（带 KV cache 复用）可以匹配同质多Agent集成的性能。只有**异构组合**（Agent 贡献结构性不同的能力）才产生一致的性能增益。

> **设计原则**: 多Agent开销的举证责任在 Harness 设计者身上。

## 十、对实践者的关键启示

### 10.1 如果你在构建 Agent 系统

- **优先投资 Harness 而非更换模型**: 证据表明 Harness 层变更的收益可比肩甚至超过模型升级
- **少即是多**: 减少工具、精简指令文件（<60行）、精选 Skill（2-3模块）通常优于更多
- **安全是架构属性**: 不能事后添加；需要从 MicroVM 隔离 + 协议层执行 + 审计日志三层防御
- **评估你测量的是什么**: 当前基准测量的是 (Agent + Harness) 对的性能，而非 Agent 本身

### 10.2 如果你在做 Agent 研究

- **报告 Harness 配置**: 每篇评估论文都应报告完整的 Harness 配置（E/T/C/S/L/V 各组件的策略）
- **做跨 Harness 实验**: AgencyBench 的方法论 — 在多个 Harness 配置中运行同一 Agent
- **关注组件耦合**: 许多重要现象只在基础设施层面才可见（跨 Harness 性能变异、评估基础设施作为误差源）
