# 项目脚手架与代码骨架

> 基于 `07_Eino_MCP_A2A_Harness_项目设计稿.md`，把架构进一步拆成可以直接开工的目录结构、接口骨架、配置文件和启动顺序。

---

## 1. 目标

这一份文档不再停留在“系统设计”层，而是回答下面几个真正开工时会遇到的问题：

- 仓库怎么分目录？
- 哪些模块先建？
- 哪些接口应该先稳定？
- 本地怎么启动？
- 第一版代码最少要写到什么程度？

我们的目标不是一步做完整个平台，而是先搭出一个 **最小但方向正确** 的骨架。

---

## 2. 第一版要交付什么

### 2.1 MVP 交付范围

- 一个 `API Gateway`
- 一个 `Gateway Agent`
- 一个 `RAG Agent`
- 一个 `Search Agent`
- 一个 `Report Agent`
- 一个最小 `MCP Tool Registry`
- 一个最小 `A2A Client`
- 一个最小 `Task State Store`
- 一个最小 `Trace / Eval` 接口

### 2.2 暂时不做什么

- 不做完整前端
- 不做复杂权限系统
- 不做任意第三方 Agent 接入
- 不做复杂工作流可视化
- 不做全量自动评估平台

## 4. 每一层为什么这样拆

### 4.1 `cmd/`

这里放进程入口。

原则：

- 每个服务一个 `main.go`
- `main.go` 只做装配，不写业务逻辑

### 4.2 `internal/agents/`

这里放每个 Agent 的核心逻辑。

建议：

- `graph.go`：Eino Graph 编排
- `state.go`：本地运行状态
- `service.go`：对外暴露的服务入口

### 4.3 `internal/mcp/`

这里不要直接等于“所有工具实现”，而是分 3 层：

- `client/`：MCP 调用能力
- `registry/`：工具可见性和校验
- `tools/`：业务层包装后的工具

### 4.4 `internal/a2a/`

这里封装 A2A 的：

- Agent Card 发现
- 任务发送
- 流式订阅
- transport 细节

这样上层业务不必直接碰 JSON-RPC/SSE 细节。

### 4.5 `internal/harness/`

这是关键层。

它不是“杂项目录”，而是项目治理层：

- hook
- policy
- eval
- trace

把这些提前独立出来，后面不会被 Agent 业务代码吞掉。

## 6. Gateway Agent 最小骨架

### 6.1 职责

Gateway Agent 第一版只做四件事：

1. 归一化问题
2. 调用 RAG Agent
3. 必要时调用 Search Agent
4. 调用 Report Agent 生成最终输出

### 6.2 状态

```go
package gateway

import "github.com/cloudwego/eino/schema"

type State struct {
    TaskID          string
    Query           string
    NormalizedQuery string
    Intent          string
    NeedSearch      bool
    InternalDocs    []*schema.Document
    ExternalDocs    []*schema.Document
    Draft           string
    Answer          string
}
```

### 6.3 Graph 伪代码

```go
func BuildGraph(ctx context.Context) (*compose.CompiledGraph, error) {
    g := compose.NewGraph[string, string](
        compose.WithGenLocalState(func(ctx context.Context) *State {
            return &State{}
        }),
    )

    g.AddLambdaNode("normalize", normalizeQuery)
    g.AddLambdaNode("call_rag", callRAGAgent)
    g.AddLambdaNode("judge_search", judgeNeedSearch)
    g.AddLambdaNode("call_search", callSearchAgent)
    g.AddLambdaNode("build_report", callReportAgent)

    g.AddEdge(compose.START, "normalize")
    g.AddEdge("normalize", "call_rag")
    g.AddEdge("call_rag", "judge_search")
    g.AddBranch("judge_search", searchBranch)
    g.AddEdge("call_search", "build_report")
    g.AddEdge("build_report", compose.END)

    return g.Compile(ctx, compose.WithMaxRunSteps(10))
}
```

### 6.4 先不要做的事情

- 不要一开始就把 Gateway 做成通用 Planner
- 不要一开始做太复杂的多路并行
- 不要让它直接知道所有底层工具

## 7. RAG Agent 最小骨架

### 7.1 职责

- 接收问题
- 调 query rewrite
- 调 retriever
- 返回候选证据

### 7.2 返回值不要直接只给答案

建议返回结构化结果：

```go
type RAGResult struct {
    Query     string
    Documents []DocumentRef
    Score     float64
}

type DocumentRef struct {
    ID      string
    Title   string
    Content string
    Source  string
}
```

这样 Gateway 后续可以：

- 做 rerank
- 做 citation build
- 判断要不要继续搜索

---

## 8. Search Agent 最小骨架

### 8.1 职责

- 对接外部搜索
- 抓取网页正文
- 清洗噪音
- 返回外部证据列表

### 8.2 第一版建议

- 先只支持文本网页
- 不做浏览器自动化
- 不做复杂交互式页面

这样风险最低，也最容易形成稳定输入。

---

## 9. Report Agent 最小骨架

### 9.1 职责

- 接收整理好的证据
- 按固定模板输出
- 返回答案和引用列表

### 9.2 输出模板建议

```text
标题
核心结论
内部资料观点
外部资料补充
差异与判断
引用列表
```

第一版不要追求花哨，追求稳定。

## 10. MCP 骨架

### 10.1 Registry 接口

```go
package registry

type Tool struct {
    Name        string
    Description string
    Scope       string
    RiskLevel   string
}

type Registry interface {
    List(scope string) []Tool
    Exists(name string) bool
    Validate(name string, args map[string]any) error
}
```

### 10.2 第一版工具范围

- `retrieve_documents`
- `rerank_documents`
- `parse_document`
- `build_citations`

这 4 个足够支撑 MVP。

### 10.3 为什么先少工具

因为工具一多就会出现：

- schema 管不过来
- agent 选择困难
- trace 难看
- hook 难做

## 13. Store 骨架

### 13.1 Repository 接口

```go
package postgres

type TaskRepository interface {
    Create(ctx context.Context, task *schema.Task) error
    UpdateState(ctx context.Context, taskID string, state schema.TaskState) error
    Get(ctx context.Context, taskID string) (*schema.Task, error)
}

type ArtifactRepository interface {
    Save(ctx context.Context, artifact *schema.Artifact) error
    ListByTask(ctx context.Context, taskID string) ([]schema.Artifact, error)

```

### 13.2 第一版建议

- Postgres 存主状态
- Redis 存短时缓存
- 如果还没准备好数据库，可以先用 memory repo 打底

这样开发不会被基础设施卡住。

### 14.3 Handler 骨架

```go
func ChatHandler(svc *task.Service) gin.HandlerFunc {
    return func(c *gin.Context) {
        var req dto.ChatRequest
        if err := c.ShouldBindJSON(&req); err != nil {
            c.JSON(400, gin.H{"error": err.Error()})
            return
        }

        resp, err := svc.HandleQuery(c.Request.Context(), req.Query)
        if err != nil {
            c.JSON(500, gin.H{"error": err.Error()})
            return
        }

        c.JSON(200, resp)
    }
}
```

## 17. 本地开发顺序

### 第一步：先跑通内存版

先不接真实 Postgres/Milvus：

- 用 memory repo
- 用 mock retriever
- 用固定 report template

目标：

- 把服务边界和调用链先跑通

### 第二步：替换真实依赖

- 把 task repo 换成 Postgres
- 把 retriever 换成 Milvus/ES
- 把 trace 接到 Langfuse

### 第三步：补 Harness 能力

- hook
- policy
- eval
- approval

顺序不要反。

## 19. 最小联调路径

建议按下面顺序联调：

### 联调 1

- API -> Gateway

确认：

- 请求能进来
- task 能创建

### 联调 2

- Gateway -> RAG Agent

确认：

- A2A 调用能成功
- 返回结构化证据

### 联调 3

- Gateway -> Report Agent

确认：

- 能把证据变成稳定输出

### 联调 4

- Gateway -> Search Agent

确认：

- 外部证据能补进来

### 联调 5

- Hook + Eval

确认：

- trace 可看
- citation 可校验

## 21. 第一版完成标志

满足下面这些，就算骨架搭成了：

- 有统一仓库结构
- 各服务有独立入口
- Gateway 能编排 RAG / Search / Report
- MCP 和 A2A 都有最小封装
- state / hook / eval 都有接口位置
- 配置和启动顺序明确

这时候项目就从“概念设计”进入“可以持续填实现”的阶段了。

## 22. 一句话总结

```text
先把骨架搭对，
再往骨架里填能力，
比先堆功能再返工稳定得多。
```

这个阶段最重要的不是“能力多”，而是：

- 目录清晰
- 边界清晰
- 接口清晰
- 启动顺序清晰

只要这四件事清楚，后面接真实模型、真实向量库、真实协议实现都会顺很多。
